## 1. Oracle DB 마이그레이션 (.169 → .184)

### 작업 목적

P6(PPM)과 연동된 Oracle XE DB를 .169 서버에서 .184 서버로 이전 후 P6 재연동 테스트

### 서버 정보

| 항목 | 내용 |
|------|------|
| 원본 서버 | 192.168.x.169 |
| 대상 서버 | 192.168.x.184 |
| Oracle 버전 | 19.3.0.0.0 Enterprise Edition |
| 인스턴스명 | unifier |

### 진행 결과

| 단계 | 상태 |
|------|------|
| .169 expdp 백업 | ✅ 완료 |
| .184로 DMP 파일 복사 | ✅ 완료 |
| .184 impdp 복원 | ✅ 완료 |
| P6 → .184 재연동 | ⏳ 미완료 |

### 복원 데이터 확인

| 테이블 | 임포트 건수 |
|--------|------------|
| PROJECT | 13건 |
| TASK | 860건 |
| TASKRSRC | 777건 |
| RSRC | 94건 |

### 핵심 복원 명령어

```shell
impdp system/<비밀번호>@unifier directory=DATA_PUMP_DIR dumpfile=CLIENT_BACKUP.DMP logfile=CLIENT_IMPORT.log full=y TABLE_EXISTS_ACTION=REPLACE
```

### 발생 에러 및 원인

| 에러코드 | 내용 | 조치 |
|----------|------|------|
| ORA-39070 | 로그 파일 열 수 없음 | DATA_PUMP_DIR 경로 재생성 |
| ORA-31684 | 객체 이미 존재함 | 무해 (스키마 기존 설치됨) |
| ORA-39083 | SYMON/DAMON 잡 존재 | 무해 |

> 에러 2043개 발생했으나 대부분 스키마 객체 충돌로 실제 데이터 복원은 정상 완료

### 배운 점

- DATA_PUMP_DIR 물리 폴더가 없으면 ORA-39070 발생 → 폴더 먼저 생성 필수
- 비밀번호에 `@` 특수문자 포함 시 따옴표 처리 방식 주의 (`\"` vs `"`)
- dbsetup은 복원과 무관 — P6 재연동 시에만 필요

## 2. PM.EXE EAccessViolation 에러

```
Exception EAccessViolation in module PM.EXE at 000000000119219C.
Access violation at address 0000000000159219C in module 'PM.EXE'.
Read of address 0000000000006A9.
```

### 원인 분석

DB 이전 후 P6 접속 설정이 여전히 .169를 바라보고 있어 연결 실패 → 크래시

### 해결 방법

- dbsetup 재실행 → .184 서버 정보로 재설정
- 또는 사용자 설정 초기화 (ADMUSER.USERSET 업데이트)

## 3. Oracle 18c XE 백업 절차서 (현대엔지니어링 고객사)

### 최종 백업 명령어

```shell
:: 1. 폴더 생성
mkdir C:\BACKUP

:: 2. SQL*Plus 접속
sqlplus /nolog
CONNECT system/"<비밀번호>"@XEPDB1

:: 3. 디렉토리 설정
CREATE OR REPLACE DIRECTORY BACKUP_DIR AS 'C:\BACKUP';
EXIT

:: 4. 백업 실행
expdp system/\"<비밀번호>\"@XEPDB1 full=y directory=BACKUP_DIR dumpfile=xedump.dmp logfile=xedump.log
```

### 핵심 주의사항

- 비밀번호에 `@` 포함 시 `\"비밀번호\"` 형식으로 이스케이프 처리 필요
- `expdp system/"<비밀번호>"@XEPDB1` → 실패  
- `expdp system/\"<비밀번호>\"@XEPDB1` → 성공
